Database Architecture (SQLAlchemy)

The backend uses SQLAlchemy ORM connected to a relational database (SQLite/PostgreSQL). The schema is designed for 3rd Normal Form (3NF) to ensure data integrity across the complex railway domain.

Core Tables & Relationships
Table NameKey ColumnsRelationships / Description
users id, email, password_hash Stores registered users. Passwords hashed via bcrypt.
stations id, code (e.g. 'MAS'), name Master list of all railway stations.
trains id, train_number, source_id, destination_id Defines a physical train. Has Foreign Keys linking to stations for origin/dest.
train_classes train_id, class_id, base_price Mapping table associating trains with available classes (e.g., 2S, CC) and their specific base fares.
train_instances train_id, journey_date, status Represents a specific run of a train on a specific date.
bookings pnr, user_id, train_instance_id, status The core reservation entity. Links a user to a specific train instance and tracks overall payment status.
passengers booking_id, name, age, seat_number Associated via cascade="all, delete-orphan" to Bookings. Stores individual traveler details.

Authentication & Security Flow

The platform uses a stateless authentication mechanism utilizing JSON Web Tokens (JWT). This ensures scalability and security without needing server-side session storage.

1
Registration API (POST /api/auth/register)
Receives user credentials. The backend utilizes passlib.context.CryptContext with the bcrypt scheme to generate a secure salt and hash before storing the password in the users table.
2
Login API (POST /api/auth/login)
Validates the plain-text password against the stored hash. If successful, it generates a JWT containing the sub (User ID) and an exp (Expiration Time) claim.
3
Token Storage & Usage
The frontend JavaScript intercepts the JWT response and stores it locally. For subsequent protected API calls (like fetching My Bookings), the token is injected into the HTTP Authorization: Bearer <token> header.
4
Route Protection
FastAPI dependencies automatically intercept the Authorization header, decode the JWT, verify the signature using the SECRET_KEY, and extract the active User ID.
Booking & Payment Workflow

The ticketing process is a multi-stage transaction to ensure that database records are only generated once secure payment confirmation is received from the Razorpay API.

1
Order Creation (POST /api/bookings/create-order)
The frontend submits the requested train, class, and passenger count. The server calculates the exact total fare (Base + Convenience + GST) and makes a secure backend HTTP call to the Razorpay /v1/orders API. It returns an order_id to the client.
2
Client-Side Checkout
The frontend initializes the Razorpay Checkout UI using the order_id. The user completes the mock payment interface. Razorpay returns a payment_id and a cryptographic signature.
3
Verification & Commit (POST /api/bookings/verify-payment)
The backend receives the signature and recalculates the expected HMAC-SHA256 hash using the RAZORPAY_KEY_SECRET.

Crucial Security Step: If hmac.compare_digest() passes, the payment is verified. Only then does the backend generate a unique PNR, create the Booking and Passenger records in the database, and return a success payload.
Secure Ticket Cancellation Logic

Ticket cancellation requires strict authorization. Instead of immediately deleting records, the system utilizes a Two-Factor Authentication (2FA) approach via email and implements "Soft Deletion" principles.

1
OTP Generation (POST /api/pnr-status/{pnr}/cancel-otp)
When a cancellation is requested, the server fetches the booking by PNR, extracts the contact_email, generates a secure 6-digit random integer, and stores it in the otp_verifications table with a 10-minute expiration timestamp.
2
SMTP Dispatch
The Python smtplib connects to a secure Gmail SMTP server to dispatch a professionally formatted HTML email containing the OTP and specific PNR details to the user.
3
Verification & State Update (POST /api/pnr-status/{pnr}/cancel-confirm)
The user submits the OTP. The server validates it against the database record and checks the expiration time.

Data Persistence: If valid, the system does not execute a SQL DELETE. Instead, it executes an UPDATE bookings SET status = 'CANCELLED'. This preserves the financial and operational history of the transaction while allowing the frontend to dynamically style the ticket in red.
